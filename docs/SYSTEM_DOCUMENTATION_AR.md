# القسم (أ) — وثيقة الأعمال (Business)

## 1. ما هو النظام؟

نظام لتتبّع طلبات مطعم **كبة زون** لحظياً، من بداية تجهيز الطلب حتى تسليمه للعميل. الفكرة الأساسية:

- كل طلب له **QR Code**.
- الموظف **يمسح الـ QR** ليُدخل الطلب للنظام، ثم يمسحه مرة أخرى عند جهوزيته.
- **شاشة عرض** كبيرة (تلفاز) تعرض الطلبات لحظياً في عمودين: **قيد التحضير** و **جاهز للاستلام** — يراها العميل والموظفون.

**القيمة:** متابعة فورية لحالة كل طلب، قياس سرعة التحضير، وتنظيم التسليم بدون فوضى.

- **اللغة:** عربية بالكامل (اتجاه RTL).
- **الفروع:** متعددة في الرياض (عرقة، ظهرة لبن، الملقا).
- **الوصول:** عبر المتصفح (ويب) — يدعم التثبيت كتطبيق (PWA).

---

## 2. المستخدمون والأدوار

| الدور | من هو | ماذا يستطيع |
|-------|-------|-------------|
| **admin (مدير)** | إدارة المطعم | كل الصلاحيات: الفروع، المستخدمون، التحليلات، السجلات، المسح، المطبخ، العرض |
| **user (موظف)** | موظف الفرع | مسح الطلبات + شاشة المطبخ فقط |
| **screen (شاشة)** | جهاز العرض (تلفاز) | شاشة العرض فقط (جلسته لا تنتهي أبداً) |

- الدخول باسم مستخدم وكلمة مرور.
- جلسة الموظف/المدير تنتهي تلقائياً بعد **12 ساعة** من عدم النشاط؛ جلسة الشاشة دائمة.

## 3. دورة حياة الطلب (الـ Workflow الكامل)

```
①  المسح الأول  ──►  ② قيد التحضير  ──►  ③ المسح الثاني (أو زر "جاهز")  ──►  ④ جاهز للاستلام  ──►  ⑤ مكتمل
   (إنشاء الطلب)        (يظهر بالشاشة)         (الموظف يعلن الجهوزية)            (يظهر بقائمة جاهز)      (تلقائي بعد 5 دقائق)
```

| المرحلة | ماذا يحدث | من ينفّذها | أين تظهر |
|---------|-----------|-----------|----------|
| ① المسح الأول | إنشاء الطلب في النظام | موظف (صفحة الماسح) | يبدأ الظهور |
| ② قيد التحضير | الطلب تحت التجهيز | — | عمود "قيد التحضير" بشاشة العرض + المطبخ |
| ③ إعلان الجهوزية | تحويل الطلب لـ "جاهز" | موظف — **بمسح ثانٍ للـ QR** أو **زر "جاهز" في المطبخ** | — |
| ④ جاهز للاستلام | الطلب جاهز للتسليم | — | عمود "جاهز" بشاشة العرض |
| ⑤ مكتمل | يُزال من الشاشة | **تلقائي** بعد ٥ دقائق من الجهوزية | يختفي من الشاشة (يبقى في التقارير) |

**نقاط مهمة:**
- **منع التكرار:** لا يمكن إنشاء نفس الطلب مرتين في نفس الفرع.
- **مهلة المسح:** ثانيتان بين كل مسحة وأخرى (منع المسح المزدوج بالخطأ).
- **الاكتمال التلقائي:** الطلب الجاهز يختفي تلقائياً بعد ٥ دقائق (قابلة للتعديل) — لا حاجة لتدخّل يدوي.
- **صوت تنبيه:** يصدر على شاشة العرض عند وصول طلب جديد.

## 4. الشاشات (من منظور الأعمال)

### 4.1 شاشة العرض (Display) — للعميل والقاعة
- شاشة تلفاز تعرض كل الطلبات النشطة في عمودين: **قيد التحضير** | **جاهز للاستلام**.
- ساعة وتاريخ مباشر + عدد الطلبات النشطة + اسم الفرع.
- صوت تنبيه عند كل طلب جديد (يمكن كتمه).
- الطلب الجاهز يختفي تلقائياً بعد المهلة.

### 4.2 صفحة الماسح (Scanner) — للموظف
- يفتح كاميرا الجهاز لمسح الـ QR.
- يعطي ردّ فوري بعد كل مسحة: **طلب جديد ✓** / **تم التجهيز ✓** / **مكرر** / **خطأ** (مع اهتزاز الجهاز).
- يعرض سجل آخر العمليات.

### 4.3 شاشة المطبخ (Kitchen) — للموظف
- تعرض الطلبات **قيد التحضير** كبطاقات، مع الوقت المنقضي لكل طلب.
- زر **"جاهز"** على كل بطاقة → بعد تأكيد، يتحول الطلب لـ "جاهز" ويختفي من المطبخ (بديل عن المسح الثاني).

### 4.4 لوحة التحليلات (Analytics) — للمدير
- **بحث برقم الطلب** في كل البيانات.
- مؤشرات: إجمالي الطلبات، متوسط وقت التحضير، أسرع/أبطأ طلب.
- رسوم بيانية: متوسط التحضير لكل فرع، الطلبات حسب الساعة.
- جدول بآخر ٥٠ طلب + **تصدير CSV**.
- فلترة بالفرع وبالفترة (اليوم / ٧ أيام / ٣٠ يوم).

### 4.5 إدارة الفروع (Admin) — للمدير
- إضافة/تعديل فرع (الاسم عربي/إنجليزي، الكود، الموقع).
- تفعيل/تعطيل فرع.
- عرض روابط كل فرع الجاهزة (شاشة العرض + الماسح).

### 4.6 إدارة المستخدمين (Add User) — للمدير
- إضافة/تعديل/حذف المستخدمين وتحديد الدور والفرع والصفحة الافتراضية.
- المدير لا يستطيع حذف حسابه.

### 4.7 سجل النظام (Logs) — للمدير
- عرض **المتصلين الآن** (المستخدمون النشطون حالياً): الاسم، الدور، الفرع، الصفحة الحالية، آخر نشاط.
- تحديث تلقائي كل ٣٠ ثانية.

## 5. سيناريوهات الأعمال (Business Cases)

| # | السيناريو | السلوك المتوقع |
|---|-----------|----------------|
| BC-1 | طلب جديد يدخل | الموظف يمسح QR → يظهر فوراً "قيد التحضير" بالشاشة + صوت تنبيه |
| BC-2 | الطلب جاهز | الموظف يمسح ثانية أو يضغط "جاهز" بالمطبخ → ينتقل لعمود "جاهز" |
| BC-3 | تسليم الطلب | يبقى بـ "جاهز" ٥ دقائق ثم يختفي تلقائياً كـ "مكتمل" |
| BC-4 | مسح نفس الطلب بالخطأ مرتين | رسالة "مكرر" — لا يُنشأ طلب مزدوج |
| BC-5 | مسح طلب جاهز بالفعل | رسالة "الطلب جاهز بالفعل" |
| BC-6 | فرع جديد | المدير يضيفه من إدارة الفروع → تظهر روابطه فوراً |
| BC-7 | موظف جديد | المدير ينشئ حساباً ويحدد دوره وفرعه |
| BC-8 | متابعة الأداء | المدير يفتح التحليلات لقياس متوسط التحضير وأبطأ/أسرع طلب |
| BC-9 | معرفة من متصل الآن | المدير يفتح "سجل النظام" |
| BC-10 | انتهاء جلسة موظف | بعد ١٢ ساعة خمول → تسجيل خروج تلقائي (الشاشة مستثناة) |

## 7. البنية العامة (Architecture)

```
المتصفح (React SPA)
   │  قراءة الطلبات (SELECT) ───────────────►  Supabase: جدول orders
   │  أي تعديل/إنشاء (عبر RPC فقط) ──────────►  Supabase: دوال RPC (SECURITY DEFINER)
   │  اشتراك لحظي (Realtime) ◄───────────────  Supabase Realtime على جدول orders
   ▼
شاشة العرض / المطبخ تتحدّث فوراً عند أي تغيير
```

**مبدأ أمني محوري:** العميل (anon) يستطيع **القراءة فقط** من الجداول. كل عمليات الإنشاء/التعديل تمرّ عبر **دوال RPC** محمية بـ `SECURITY DEFINER` وتتحقق من جلسة المستخدم ودوره. لا توجد كتابة مباشرة من المتصفح.

## 8. قاعدة البيانات (Schema)

### `branches` — الفروع
`id, name_ar, name_en, code (UNIQUE), location_label, is_active, created_at`

### `orders` — الطلبات
`id, order_id, branch_id→branches, channel_link, status('preparing'|'ready'|'completed'), scanned_at, ready_at, completed_at, prep_duration_seconds (GENERATED = ready_at − scanned_at), raw_qr_data (JSONB), created_at`
قيد فريد: `UNIQUE(order_id, branch_id)`.

### `scan_logs` — سجل المسح
`id, order_id→orders, scan_type('first_scan'|'second_scan'), scanned_by, scanned_at, device_info`

### `users` — المستخدمون
`id, username (UNIQUE), password (مشفّر bcrypt عبر trigger), branch_id, route, role('admin'|'user'|'screen'), created_at`

### `user_sessions` — الجلسات
`id, user_id→users, created_at, expires_at (افتراضي +12 ساعة)`

### `active_sessions` — الحضور اللحظي
`id, user_id (UNIQUE), current_page, last_heartbeat, started_at` — يُحدّث كل ٣٠ ثانية لتتبّع "المتصلين الآن".

> **RLS:** `orders` و `scan_logs` و `branches` → قراءة عامة فقط. `users` و `user_sessions` و `active_sessions` → مغلقة تماماً (الوصول عبر RPC فقط). **Realtime مفعّل على `orders` فقط.**

## 9. دوال RPC الرئيسية

| الدالة | الوظيفة | الصلاحية |
|--------|---------|----------|
| `authenticate_user` | الدخول + إنشاء جلسة + إرجاع `sessionId` | عام |
| `get_session_user` | التحقق من الجلسة وإرجاع (user_id, role, branch_id) | داخلي |
| `rpc_scanner_insert_order` | المسح الأول — إنشاء الطلب (preparing) + log | admin/user |
| `rpc_scanner_mark_ready` | المسح الثاني / زر المطبخ — تحويل لـ ready + log | admin/user |
| `rpc_display_complete_order` | إكمال الطلب الجاهز (تلقائي من الشاشة) | admin/screen |
| `rpc_admin_upsert_branch` / `rpc_admin_toggle_branch` | إدارة الفروع | admin |
| `rpc_create_user_secure` / `rpc_delete_user_secure` / `rpc_list_users_secure` / `rpc_update_user_secure` | إدارة المستخدمين | admin |
| `rpc_upsert_heartbeat` / `rpc_remove_presence` / `rpc_get_online_users` | تتبّع الحضور اللحظي | حسب الدور |

## 11. آليات تقنية مهمة

- **التحديث اللحظي (Realtime):** `useOrders` يشترك في تغييرات جدول `orders` المفلترة بالفرع (INSERT/UPDATE/DELETE) فيتحدّث العرض فوراً بدون إعادة تحميل.
- **منطق المسح (`useScanner`):** يقرأ الـ QR (JSON أو نص عادي) → إن لم يوجد الطلب: إنشاء (المسح الأول) → إن كان "preparing": تحويل لـ ready (المسح الثاني) → غير ذلك: "مكرر/جاهز". مهلة ثانيتين بين المسحات.
- **المصادقة:** مبنية على قاعدة البيانات (ليست Supabase Auth) — كلمات المرور مشفّرة bcrypt عبر trigger، والتحقق عبر RPC. الجلسة محفوظة في `localStorage` بمفتاح `kz_session` ومتزامنة بين التابات.
- **الخمول:** `useIdleTimer` ينهي جلسة user/admin بعد ١٢ ساعة خمول (يتجاهل دور screen).
- **الحضور:** `useHeartbeat` يرسل نبضة كل ٣٠ ثانية لتحديث `active_sessions` (تُستخدم في صفحة السجلات).
- **تحديد القناة:** من `channel_link` → "جاهز" (jahez) / "هنقرستيشن" (hungerstation) / "مباشر".
- **الاكتمال التلقائي:** شاشة العرض تستدعي `rpc_display_complete_order` بعد مرور `VITE_READY_TIMEOUT_MINUTES` على جهوزية الطلب.

## 14. ملاحظات وحدود حالية

- صيغة الـ QR تدعم JSON (`{order_id, channel_link, location}`) أو نصاً عادياً يُعامَل كرقم طلب.
- سياسات RLS الحالية: قراءة عامة + كتابة عبر RPC فقط (نموذج MVP آمن).
- لا يوجد ربط آلي بنظام نقاط البيع بعد — إدخال الطلبات يدوي بالمسح. (**تكامل Foodics مُجهّز وغير مفعّل** — راجع خطة 2026-06-21.)
- كل صفحة لها ملف CSS مستقل بجانب Tailwind.
